# 🔍 Collabryx Database Seeder — Audit Report

**Date:** 2026-05-31  
**Scope:** `scripts/seed-data/` — All seeder modules, config, data generators, and DB schema coverage  
**Audit Type:** Gap analysis — tables seeded vs tables missing

---

## Executive Summary

The seeder system at `scripts/seed-data/` currently handles **~16 of 31+ database tables** via 8 active seeder modules + 3 legacy modules. **18+ tables are completely unseeded**, including match preferences, post attachments, comment likes, analytics aggregates, events, audit logs, moderation logs, feed scores, and more. The `__init__.py` is outdated (references 3 legacy monolithic seeders that no modules use). The interactive menu and config need extension to cover all new modules.

## 1. 🏗 Architecture Issues

### 1.1 Outdated `seeders/__init__.py`
**File:** `scripts/seed-data/seeders/__init__.py`

```python
__all__ = [
    "profiles_seeder",
    "content_seeder",      # LEGACY - replaced by posts_seeder
    "social_seeder",        # LEGACY - replaced by connections_seeder + matches_seeder
    "messaging_seeder",     # LEGACY - replaced by conversations_seeder + messages_seeder
    "notifications_seeder",
    "mentor_seeder",
    "embeddings_seeder",
]
```

**Issues:**
- References 3 legacy monolithic seeders that are no longer used
- Missing all new atomic seeders: `posts_seeder`, `connections_seeder`, `matches_seeder`, `conversations_seeder`, `messages_seeder`
- The interactive menu bypasses `__init__.py` entirely (imports classes directly)

**Fix:** Remove legacy references. Add all active atomic modules to `__all__`.

---

### 1.2 Two Competing Seeder Patterns

**Pattern A (Standalone):** `profiles_seeder.py` — does NOT inherit from `BaseSeeder`, uses raw `httpx.Client` directly  
**Pattern B (Inheritance):** All other seeders inherit from `BaseSeeder` and use `self.create_single()` / `self.http`

**Issue:** Inconsistency. New seeders must choose which pattern to follow. Profiles seeder is standalone because it hits the Auth admin API (`/auth/v1/admin/users`). All other tables use `/rest/v1/`.

**Fix:** Documented pattern — new seeders for regular tables should always inherit `BaseSeeder`. Only auth-related operations should use standalone pattern.

---

### 1.3 BaseSeeder Cache Limitations

**File:** `scripts/seed-data/seeders/base_seeder.py`

The `BaseSeeder` class has `_user_ids_cache` and `_existing_data_cache` dictionaries, but:
- Only caches `profiles`, `posts`, `conversations`, `connections` data
- Doesn't cache many tables the new seeders will need (e.g., `comments` for comment_likes, `match_suggestions` for match_scores)
- Each new seeder re-implements its own duplicate checking cache

**Fix:** Consider adding generic `fetch_existing_ids(table_name, id_field)` method to `BaseSeeder`.

## 3. ⚠️ Critical Issues

### 3.1 Seeders Don't Create Privacy/Theme/Notification Prefs

These three tables have **database triggers** (`handle_new_privacy_settings()`, `handle_new_theme_preferences()`, `handle_new_notification_preferences()`) that auto-create rows when a profile is inserted via the Supabase dashboard or app. **However**, the REST API seeder may bypass these triggers depending on how the POST is handled.

**Action:** Verify trigger behavior for REST API inserts. If triggers fire, these tables are covered. If not, create dedicated seeders.

### 3.2 Events Table is Auto-Populated

The `events` table has **8 event capture triggers** that fire from `post_reactions`, `comments`, `connections`, `messages`, `match_activity`, `posts`, and `profiles`. If these triggers work with REST API inserts, events are auto-populated.

**Action:** Verify trigger behavior. If events are auto-generated, a dedicated events seeder is lower priority. If not, seed ~500 synthetic events.

### 3.3 Auth Users Reference

`audit_logs.user_id` references `auth.users`, NOT `public.profiles`. This means the seeder must either:
1. Fetch user IDs from the auth admin API (`/auth/v1/admin/users`)
2. Or skip the `user_id` field (it's SET NULL)

**Action:** Add a note to the audit_logs seeder to handle this correctly.

### 3.4 Aggregate Tables Need Plausible Data

`user_analytics` and `platform_analytics` are pure aggregate tables. They don't need to match real computed values — but they should look realistic:

- Platform analytics: 7-30 days of historical data with trending patterns
- User analytics: per-user metrics correlated with their actual profile data

## 7. 🔧 Proposed Seeder Template

For every new seeder, follow this pattern:

```python
"""
{Entity} Seeder
Creates {entity} data using Supabase REST API
"""
import random, time
from typing import List, Dict, Any, Optional
from colorama import Fore, Style
from config import config
from seeders.base_seeder import BaseSeeder

class {Entity}Seeder(BaseSeeder):
    """Seeder for {entity} with incremental seeding"""

    def __init__(self, http_client):
        super().__init__(http_client)
        self.stats = {"created": 0, "skipped": 0, "failed": 0}
        self._cache = set()

    def _load_cache(self):
        """Load existing records from DB"""
        try:
            response = self.http.get(
                f"{config.SUPABASE_REST_URL}/{table}?select=id{unique_field}",
                headers=config.API_HEADERS,
            )
            response.raise_for_status()
            records = response.json() or []
            self._cache = {r[unique_field] for r in records}
        except Exception as e:
            print(f"{Fore.RED}✗ Failed to load cache: {e}{Style.RESET_ALL}")

    def seed(self, limit: Optional[int] = None) -> Dict[str, int]:
        self.stats = {"created": 0, "skipped": 0, "failed": 0}
        # ... load cache, loop, create_single, rate limit, stats ...
        return self.stats
```

### ❌ CRITICAL BUG: `capture_event()` Still Broken in Master Setup

**Location:** `99-master-all-tables.sql`, lines 2909-2952

```sql
-- BROKEN: evaluates ALL COALESCE args eagerly
COALESCE(NEW.user_id, NEW.author_id, NEW.actor_user_id, NEW.requester_id, NEW.sender_id)
```

**Root cause:** PostgreSQL evaluates all arguments of `COALESCE()` before picking the first non-null. When this trigger fires on:
- **`comments` table** — `NEW.user_id` doesn't exist (column is `author_id`) → crashes with `"record \"new\" has no field \"user_id\""`
- **`post_reactions` table** — `NEW.author_id` doesn't exist → same crash

**Fix exists** in `supabase/migrations/20260531_fix_capture_event.sql` but the master file was **never updated** to incorporate it. The migration uses a `CASE TG_TABLE_NAME` branch instead.

**Impact:** Events for `post_reactions`, `comments`, `connections`, `messages`, and `match_activity` silently fail to insert into `events` table — breaking analytics.

---

### ❌ RACE CONDITION: `check_embedding_rate_limit()` Not Atomic

**Location:** `99-master-all-tables.sql`, lines 1344-1374

```sql
SELECT * INTO v_record FROM public.embedding_rate_limits ...  -- read
-- ... later ...
UPDATE public.embedding_rate_limits SET request_count = request_count + 1 ... -- write
```

**Problem:** `SELECT` then `UPDATE` is **not atomic**. Under concurrent load (two onboarding flows simultaneously), both can read `request_count = 1`, both pass the limit check, and both increment — allowing up to 6 requests instead of 3/hour.

**Fix:** Use `SELECT ... FOR UPDATE` or atomic `UPDATE ... RETURNING`.

---

### ❌ INCONSISTENCY: `posts.version` Not in CREATE TABLE

**Location:** `99-master-all-tables.sql`, lines 134-148 vs lines 700-705

The `posts` CREATE TABLE does **not** include `version`. It's added later via:

```sql
ALTER TABLE public.posts ADD COLUMN IF NOT EXISTS version INTEGER NOT NULL DEFAULT 1;
```

**Problem:** The `posts_bump_version()` trigger references `OLD.version`/`NEW.version` — if the ALTER TABLE is somehow skipped, the trigger crashes. Should be in the CREATE TABLE directly.

---

### ❌ `events.event_category` — GENERATED Column vs DEFAULT

| Reference Schema | Master Setup |
|-----------------|--------------|
| `DEFAULT CASE WHEN event_type LIKE 'post_%' THEN 'content' ... END` | `GENERATED ALWAYS AS (CASE ... END) STORED` |

**Why the master is correct:** A `DEFAULT` can be overridden at insert time, allowing `event_type = 'post_created'` with `event_category = 'network'` — breaking analytics. `GENERATED ALWAYS AS` guarantees consistency.

### 📝 REFERENCE SCHEMA SYNTAX ERRORS

The reference schema uses invalid PostgreSQL syntax in 3 places:

```sql
-- ❌ Will throw syntax error:
interested_in_types ARRAY DEFAULT '{}'::text[]
overlapping_skills ARRAY DEFAULT '{}'::text[]
shared_interest_tags ARRAY DEFAULT '{}'::text[]

-- ✅ Must be:
interested_in_types TEXT[] DEFAULT '{}'
```

`ARRAY` is **not a valid PostgreSQL column type**. Only `TEXT[]`, `INTEGER[]`, etc. are valid.
